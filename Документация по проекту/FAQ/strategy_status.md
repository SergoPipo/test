# Статус стратегии — что значит, как меняется, кто меняет

> **Версия:** 1.0 · **Дата создания:** 2026-04-28 · **Источник:** аудит после жалобы пользователя «не вижу возможности изменить статус».
>
> Документ описывает все статусы стратегии, их смысл, текущие правила переходов между ними, и где (не) есть UI для смены.

---

## 1. Два разных «статуса» — не путать

В системе используются **два разных понятия статуса**, которые часто путаются:

### 1.1 Статус **стратегии** (`Strategy.status`)

Хранится в БД в колонке `strategies.status`. Это **маркер от пользователя**, описывающий стадию жизненного цикла стратегии.

Возможные значения (`VALID_STATUSES` в [`app/strategy/schemas.py:7`](../../Develop/backend/app/strategy/schemas.py)):

| Значение | Лейбл UI | Цвет | Смысл |
|----------|----------|------|-------|
| `draft` | Черновик | gray | Стратегия в разработке, не проверена |
| `tested` | Протестирована | blue | По стратегии прогнаны backtests, но в торговле не запущена |
| `paper` | Paper Trading | yellow | В виртуальной торговле (без реальных денег) |
| `live` | Real Trading | red | В реальной торговле через брокера |
| `paused` | Пауза | orange | Временно приостановлена (live или paper не идут) |
| `archived` | Архив | gray | Снята с активной разработки, не отображается в основном списке |

### 1.2 Статус **тикера в стратегии** (`StrategyInstrumentSummary.status`)

Вычисляется на лету в [`app/strategy/service.py:185-198`](../../Develop/backend/app/strategy/service.py) для каждого тикера, привязанного к стратегии (через бэктесты или активные сессии). **Не хранится** в БД — пересчитывается при каждом GET.

Алгоритм:

```python
if session is not None:
    if session.mode == "paper":
        status = "paper"
    elif session.mode == "real":
        status = "real"
    elif session.mode == "sandbox":
        status = "paper"   # sandbox показываем как paper
elif backtest is not None:
    status = "tested"
else:
    status = "draft"
```

Возможные значения тикера: `draft, tested, paper, real, paused` (5; `archived` не применимо к тикеру).

⚠️ **Несогласованность:** статус стратегии использует `live`, а статус тикера — `real`. Это исторический drift — UI отображает оба как «Real Trading», но это разные строки. Кандидат на унификацию в S8 (но не баг — работает).

### 1.3 Где они показываются

| Место | Что показывает |
|-------|-----------------|
| Колонка «Статус» в таблице **стратегий** на dashboard | `Strategy.status` (ручной от пользователя) |
| Колонка «Статус» в раскрытой строке **тикера** | `StrategyInstrumentSummary.status` (автомат) |
| Фильтры над таблицей (`Все / Черновик / Активные / Архив`) | По `Strategy.status` |

## 2. Текущие правила перехода между статусами `Strategy.status`

### 2.1 Создание стратегии → `draft`

При создании стратегии (`POST /api/v1/strategies`) status **всегда** ставится в `draft` ([`app/strategy/service.py:42`](../../Develop/backend/app/strategy/service.py)). Нельзя создать стратегию сразу с `paper` или `tested`.

### 2.2 Изменение статуса — только вручную через API

Статус изменяется только через `PATCH /api/v1/strategies/{id}` с body `{"status": "<новое значение>"}` ([`app/strategy/service.py:261-285`](../../Develop/backend/app/strategy/service.py)).

Backend валидирует значение против `VALID_STATUSES`, но **не проверяет логику переходов** — можно перейти из `draft` сразу в `archived`, из `live` в `draft`, и т.д. Никаких state-machine ограничений нет.

### 2.3 🔴 Нет UI для смены статуса

В текущей версии (Sprint 7 baseline) **в интерфейсе нет ни одного места**, где пользователь мог бы изменить статус стратегии. Конкретно проверено:

- [`pages/StrategyEditPage.tsx`](../../Develop/frontend/src/pages/StrategyEditPage.tsx) — есть `useState('draft')` для статуса, но нет ни `<Select>`, ни кнопки «В архив», ни «Запустить paper» / «Перейти в live». Status подгружается из БД и только отображается, не редактируется.
- [`pages/DashboardPage.tsx`](../../Develop/frontend/src/pages/DashboardPage.tsx) — таблица стратегий, статус только отображается.
- Других страниц редактирования стратегии нет.

Единственный способ сейчас сменить статус — прямой `curl`/Postman:

```bash
curl -X PATCH http://localhost:8000/api/v1/strategies/<ID> \
  -H "Authorization: Bearer <TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{"status": "tested"}'
```

**Это техдолг.** Карточка `S7R-STRATEGY-STATUS-CHANGE-UI` (medium-high) в S8 backlog — добавить UI для управления статусом стратегии.

### 2.4 Автоматических переходов **нет**

В коде отсутствует автоматическая логика типа «когда первый бэктест завершён → status `draft → tested`» или «когда стартовала live-сессия → `tested → live`». Бэктесты и live-сессии живут параллельно, не влияя на `Strategy.status`.

Это **сознательное решение** (S5 design): пользователь должен сам решать, на какой стадии находится его стратегия. Статус — это **семантический маркер**, не вычисляемый.

⚠️ **Однако:** колонка «Статус» в таблице стратегий **всегда показывает `draft`**, потому что:
- Backend при create ставит `draft` хардкодом.
- Никогда не меняет автоматически.
- UI не даёт сменить вручную.

⇒ **Все стратегии у всех пользователей сейчас в статусе `draft`** (если не патчили БД руками).

### 2.5 Каскадные эффекты

Изменение статуса стратегии **не влияет** на:
- Связанные `trading_sessions` (продолжают работать как раньше).
- Связанные `backtests` (история сохраняется).
- `current_version_id` (текущая версия не меняется).
- `live_trades` (открытые позиции не закрываются).

То есть `archived` — это **только маркер видимости в фильтре**, а не «остановка торговли». Если у архивированной стратегии есть активная live-сессия — она продолжит торговать, просто стратегия будет в фильтре «Архив» вместо «Активные».

## 3. Что показывает каждый фильтр на dashboard

```
[Все (n)] [Черновик (n)] [Активные (n)] [Архив (n)]
```

Логика в [`pages/DashboardPage.tsx:61-72`](../../Develop/frontend/src/pages/DashboardPage.tsx):

| Фильтр | SQL-эквивалент |
|--------|----------------|
| **Все** | `WHERE 1=1` (все стратегии пользователя) |
| **Черновик** | `WHERE status = 'draft'` |
| **Активные** | `WHERE status IN ('paper', 'live', 'tested')` |
| **Архив** | `WHERE status = 'archived'` |

⚠️ Статус `paused` **не попадает ни в один фильтр** — стратегии в этом статусе видны только во «Все» (известный нюанс, кандидат на правку в S8).

## 4. Что хочется сделать в S8

Карточка `S7R-STRATEGY-STATUS-CHANGE-UI` (medium-high):

1. **На странице `/dashboard` в строке стратегии (контекстное меню `⋮`)** — добавить пункты:
   - «Архивировать» (только если не `archived`) → PATCH status=`archived`
   - «Восстановить из архива» (только если `archived`) → PATCH status=`draft`
   - «Поставить на паузу» (только если `paper`/`live`) → PATCH status=`paused`
   - «Продолжить» (только если `paused`) → PATCH status=`paper`/`live` (запоминать предыдущий)

2. **На странице `/strategy/{id}/edit`** — добавить `<Select>` со списком всех статусов сверху (рядом с именем стратегии).

3. **(Опционально)** Автопереход `draft → tested` при первом успешном бэктесте — обсуждается.

4. **(Опционально)** Автопереход `tested → paper` при первом запуске paper-сессии — обсуждается.

Какие из автопереходов реализовать — отдельный UX-вопрос для брейншторма перед S8.

## 5. FAQ — частые вопросы

### Я создал стратегию, она в «Черновике». Как перевести в «Paper Trading»?

В текущей версии — никак через UI. Только через `PATCH /api/v1/strategies/<ID>` с `{"status": "paper"}`. UI добавим в S8 (`S7R-STRATEGY-STATUS-CHANGE-UI`).

### У меня запущена paper-сессия, но стратегия в статусе «Черновик». Это нормально?

Сейчас — да. Статус стратегии не меняется автоматически при запуске сессии. **Статус тикера** в раскрытой строке покажет «Paper» (вычисляется из live-сессии), а **статус стратегии в основной строке** останется «Черновик», пока пользователь не сменит вручную.

### Можно архивировать стратегию, у которой есть активная live-сессия?

Технически **да** — backend не блокирует. Но live-сессия продолжит торговать. Это потенциально опасно (пользователь думает «я заархивировал, ничего не работает», а на деле — работает). Рекомендация для S8: при попытке архивирования стратегии с активными сессиями показывать confirm-modal «У вас есть активные сессии: ... Они продолжат работать. Остановить их?».

### Почему статус «Активные» включает не только `paper`/`live`, но и `tested`?

Это договорённость UX: «Активные» = «над чем сейчас идёт работа» (включая прогон бэктестов). `tested` означает «есть результаты бэктестов, но не запущена в торговлю» — пользователь активно с ней работает, скоро запустит.

### Что значит `paused`?

Это статус для случаев, когда пользователь временно остановил стратегию, но не хочет её архивировать. Например, во время отпуска или ребалансировки. Сейчас не используется автоматически (Circuit Breaker НЕ переводит стратегию в `paused`, он переводит **сессию** в `status='paused'` — это разные сущности).

### Статус стратегии vs статус сессии — это одно и то же?

**Нет.** Это две разные сущности:
- **Strategy.status** — маркер от пользователя на уровне стратегии (`draft, tested, paper, live, paused, archived`).
- **TradingSession.status** — статус конкретной торговой сессии (`active, paused, stopped`). Управляется автоматически Trading Engine + Circuit Breaker. Может быть несколько сессий по одной стратегии (одна на тикер).

См. также `app/trading/models.py:37` — `TradingSession.status` enum.
