## 1. Обзор системы

### 1.1 Назначение и цели

Веб-приложение, предоставляющее трейдеру инструмент для:
- описания торговых стратегий на естественном языке с помощью AI-ассистента;
- тестирования стратегий на исторических данных (бэктест) с визуализацией результатов;
- запуска стратегий в режиме бумажной (виртуальной) и реальной торговли через брокерские API;
- мониторинга сделок и управления позициями через веб-интерфейс и Telegram-бота.

### 1.2 Целевая аудитория

Частный трейдер, понимающий процессы торговли финансовыми инструментами, но не занимающийся этим профессионально. Пользователь не обязан иметь навыки программирования — система предоставляет AI-помощника для создания стратегий.

### 1.3 Фаза 1: ограничения первой версии

- Локальный deployment на персональном компьютере пользователя.
- Один зарегистрированный пользователь.
- Desktop-first интерфейс (мобильная версия не предусмотрена).
- Нет публичного API для внешних интеграций.
- Только длинные позиции (long). Short-ордера блокируются на уровне валидации, но архитектура проектируется с учётом активации Short в фазе 2.
- Accessibility (WCAG) не входит в scope фазы 1.
- При выходе на рынок (фаза 2) архитектура должна поддерживать переход к мультипользовательскому режиму без переработки ядра.

### 1.4 Поддерживаемые рынки и инструменты

| Рынок | Инструменты |
|---|---|
| Фондовый рынок MOEX | Акции (обыкновенные и привилегированные) |
| Долговой рынок | ОФЗ, корпоративные облигации |
| Валютный рынок / коллективные инвестиции | Валютные пары, ETF, БПИФ |

> **Примечание:** Срочный рынок FORTS (фьючерсы) не поддерживается в фазе 1. Архитектура брокерского адаптера предусматривает расширение на фьючерсы в фазе 2 (включая учёт ГО, вариационной маржи, клирингов).

### 1.5 Поддерживаемые таймфреймы

`1м`, `5м`, `15м`, `1ч`, `4ч`, `Д`, `Н`, `М`

### 1.6 Торговые сессии и расписание MOEX

Система загружает расписание торговых сессий динамически из MOEX ISS API при старте и автоматически обновляет ежедневно. Ниже приведены значения по умолчанию (fallback при недоступности API):

| Рынок | Основная сессия (MSK) | Аукционы |
|---|---|---|
| Фондовый рынок | 10:00–18:39:59 | Аукцион открытия 09:50–10:00, аукцион закрытия 18:40–18:50 |
| Долговой рынок | 10:00–18:39:59 | Аукцион открытия 09:50–10:00 |
| Валютный рынок | 10:00–23:49:59 | — |

Требования:
- Расписание загружается из MOEX ISS API при старте системы и обновляется автоматически раз в сутки. При недоступности API используются fallback-значения из таблицы выше.
- Выставление ордеров вне торговых часов **блокируется** с уведомлением пользователю.
- Тайм-фильтр в стратегии (блок «Тайм-фильтр») учитывает расписание сессий: пользователь задаёт часы торговли только в пределах активной сессии.
- В бэктесте: сигналы, сгенерированные вне торговых часов, игнорируются.

### 1.7 Торговый календарь MOEX

- Загрузка и кеширование календаря нерабочих и праздничных дней с MOEX ISS API.
- Обновление календаря: при старте системы и автоматически раз в сутки.
- Стратегии и бэктесты учитывают нерабочие дни — в эти дни сигналы не генерируются, ордера не выставляются.
- При запуске бэктеста: период автоматически корректируется с исключением нерабочих дней (нет искусственных пропусков в данных).

### 1.8 Локализация и форматы

- Язык интерфейса: русский. Архитектура интернационализации (i18n) не требуется в фазе 1.
- Формат даты: ДД.ММ.ГГГГ.
- Формат чисел: разделитель тысяч — пробел, десятичный — запятая (например: 1 234 567,89).
- Валюта: ₽ (российский рубль).

## 2. Пользовательские роли и авторизация

### 2.1 Регистрация и вход

- Первый запуск: мастер начальной настройки создаёт единственного пользователя (логин + пароль).
- Вход по логину + пароль.
- Восстановление пароля: через CLI-утилиту (локальное приложение, SMTP не требуется).
- Полноценная email-based авторизация (регистрация по email, восстановление пароля по email, 2FA) — фаза 2.

> **Примечание для ТЗ:** Пароли хранятся с использованием безопасного хеширования. Защита от брутфорса: блокировка после 5 неудачных попыток на 15 минут. Сессии реализуются через токены с ротацией. Конкретные алгоритмы и параметры определяются в ТЗ.

### 2.2 Профиль пользователя

- Редактирование пароля.
- Указание email для уведомлений (используется только для отправки уведомлений, не для авторизации).
- Управление подключёнными брокерскими аккаунтами (добавление, редактирование, удаление).
- Привязка Telegram-аккаунта через уникальный одноразовый токен (пользователь вводит токен в бота, бот регистрирует chat_id).
- Настройки уведомлений: выбор каналов (Telegram / email / in-app) и событий.
- Настройки риск-менеджмента (параметры используются механизмом circuit breaker, см. раздел 12.4):
  - Дневной лимит убытков (% от баланса или фиксированная сумма).
  - Максимальная просадка от пика (% от максимума equity).
  - Максимальный размер одной позиции.
  - Лимит на количество сделок в день (по всем стратегиям суммарно).

## 3. Управление стратегиями

### 3.1 Главный дашборд [Must]

Таблица всех стратегий пользователя с колонками:

| Колонка | Описание |
|---|---|
| Название стратегии | Задаётся пользователем |
| Статус | Черновик / Протестирована / Бумажная торговля / Боевая / Пауза / Архив |
| Инструменты | Перечень тикеров, на которых выполнялись бэктесты или запущена торговля |
| Последний бэктест | Дата, таймфрейм, ключевые метрики (профит-фактор, просадка) |
| Боевой режим | Брокер, тикер, режим (Paper / Real), текущий P&L |

Действия из дашборда: создать стратегию, открыть стратегию, запустить бэктест, запустить торговлю.

**Фильтр по статусам (S5+):** чипы «Все / Черновик / Протестированы / Активные / Архив» с подсчётом количества стратегий в каждой группе. `latest_bt_by_ticker` фильтрует по `bt_row.status == "completed"` — failed-бэктесты не попадают в колонку «Последний бэктест».

### 3.2 Создание и редактирование стратегии [Must]

Страница редактирования стратегии (`/strategies/:id`) содержит две вкладки: **«Содержание»** и **«Редактор»**.

**Вкладка «Содержание»:**
- Описание стратегии (textarea, autosize) — текстовое описание в свободной форме.
- Параметры стратегии (стоп-лосс, тейк-профит, размер позиции, макс. позиций).

**Вкладка «Редактор»:**
Визуальный блочный редактор (Google Blockly) с кастомным toolbox (Mantine Accordion). Два режима:

- **Режим Blockly (по умолчанию):** блочный workspace слева, панель описания справа (SharedDescriptionPanel). Доступные типы блоков:
  - «Индикатор» (SMA, EMA, RSI, MACD, Bollinger Bands, ATR, Stochastic, CCI, Volume)
  - «Условие» (пересечение/Crossover, больше/меньше/Compare, нахождение в зоне/InZone, таймфрейм)
  - «Логический оператор» (И, ИЛИ, НЕ) — сигналы поддерживают динамическое количество условий (N входов)
  - «Сигнал входа» / «Сигнал выхода»
  - «Тайм-фильтр» (торговать только в определённые часы / дни)
  - «Управление капиталом» (% от депозита, стоп-лосс, тейк-профит)

- **Режим AI-помощник:** полноэкранный режим со свайп-анимацией (300ms). Split layout: описание стратегии (40%) слева + AI-чат (60%) справа с resizable разделителем. AI помогает сформировать описание в формате шаблона, которое затем конвертируется в блоки через Template Parser.

Кнопка «AI-помощник» переключает между режимами. Кнопка «Применить на схеме» — парсит описание в блоки и возвращает к Blockly. Кнопка «Заполнить по схеме» — генерирует описание из текущих блоков.

**Генерация кода:**
Кнопка «Генерировать» генерирует Python-код для Backtrader по блочной схеме. Код отображается в панели для ознакомления. Кнопка неактивна (disabled) если на workspace нет блоков.

### 3.3 Версионирование стратегий [Should]

- Каждое сохранение изменений в описании, блоках или коде создаёт новую версию.
- Версии нумеруются последовательно (v1, v2, …).
- Результаты бэктестов привязаны к конкретной версии кода — изменение стратегии не инвалидирует старые результаты.
- UI: история версий в боковой панели с возможностью просмотра и отката.

### 3.4 Параметры стратегии [Must]

- Выбор режима расчёта размера позиции: **% от депозита** (например, 10% на сделку) или **приблизительная сумма** (например, 50 000 ₽ ± допустимое отклонение на основе цены лота).
- **Лотность:** система автоматически округляет размер позиции до целого числа лотов, запрашивая размер лота инструмента через брокерский API. При расчёте «приблизительной суммы» итоговый объём = ближайшее целое число лотов, не превышающее заданную сумму.
- Стоп-лосс и тейк-профит: задаются в блоках стратегии или как глобальные параметры.
- Максимальное количество одновременно открытых позиций по данной стратегии.

### 3.5 Изоляция стратегий [Must]

- Каждая запущенная стратегия (стратегия + инструмент + брокер) ведётся в **отдельном дашборде** с собственным P&L, историей сделок и статистикой.
- Сделки, открытые в рамках одной стратегии, **не пересекаются** с позициями другой стратегии: система ведёт виртуальный учёт позиций per-strategy, даже если на уровне брокера позиция общая (нетто).
- **Ограничение в режиме Real:** запрещено одновременно запускать две стратегии на одном инструменте в одном направлении на одном брокерском счёте. При попытке — блокировка с предупреждением: «На инструменте {TICKER} уже запущена стратегия {NAME} в режиме Long».
- **Ограничение на разнонаправленную торговлю:** см. раздел 12.4 (circuit breaker).
- В режиме Paper — допускается параллельный запуск нескольких стратегий на одном инструменте (виртуальные портфели независимы).

## 4. Бэктестинг

### 4.1 Настройка запуска бэктеста [Must]

Параметры, задаваемые пользователем перед запуском:

| Параметр | Описание |
|---|---|
| Финансовый инструмент | Тикер (один или несколько для портфельного бэктеста) |
| Период | Дата начала — дата окончания (до 5 лет истории) |
| Таймфрейм | Один из поддерживаемых (1м … М) |
| Комиссия брокера | % от оборота или фиксированная сумма на сделку |
| Проскальзывание | % от цены или фиксированное количество пунктов |
| Начальный капитал | Сумма в рублях для расчёта P&L |

> **Примечание:** Модель проскальзывания в фазе 1 — упрощённая (фиксированный % или пункты). В фазе 2 планируется Volume Impact Model (учёт объёма заявки и ликвидности стакана).

**Ограничение на делистинговые инструменты:** бэктест доступен только для инструментов, торгующихся на MOEX на текущий момент. Статус инструмента проверяется через брокерский API / MOEX ISS перед запуском. Если инструмент делистингован или приостановлен — бэктест блокируется с сообщением: «Инструмент {TICKER} не торгуется на MOEX. Бэктест недоступен».

### 4.2 Логика работы с историческими данными [Must]

```
Запрос бэктеста
    ↓
Проверка кеша в БД (инструмент + таймфрейм + период)
    ↓ есть           ↓ нет (или частично)
Используем БД → Загружаем недостающее от брокера / MOEX API
                            ↓
                   Сохраняем в БД (дополнение)
                            ↓
                   Валидация данных
                            ↓
                   Передаём в Backtrader
```

- Приоритетный источник данных: T-Invest API; резервный — MOEX ISS (бесплатное публичное API Московской биржи).
- Формат хранения в БД: OHLCV (open, high, low, close, volume) с временной меткой и таймфреймом.

**Валидация данных:** перед запуском бэктеста система проверяет загруженные OHLCV-данные:
- Отсутствие пропусков (gap > 1 свеча на данном таймфрейме в торговые дни с учётом торгового календаря).
- Отсутствие аномальных значений (OHLC вне допустимого диапазона: например, High < Low, Close за пределами ±50% от предыдущего Close).
- Отсутствие дубликатов по временной метке.
- При обнаружении проблем — предупреждение пользователю с деталями и возможностью продолжить или отменить бэктест.

**Корпоративные действия:** система отображает предупреждение, если в периоде бэктеста для инструмента зафиксированы сплиты или крупные дивиденды (информация от MOEX ISS). Рекомендация пользователю: использовать скорректированные цены (adjusted).

### 4.3 Оптимизация параметров [Should]

- Grid search: пользователь задаёт диапазон и шаг для числовых параметров стратегии (например, период EMA от 10 до 50 с шагом 5).
- Результат: таблица со всеми комбинациями параметров и метриками — сортировка по любому показателю.
- Предупреждение об overfitting: система информирует пользователя, если лучший результат достигнут на очень узком наборе параметров.

### 4.4 Портфельный бэктест [Could]

- Одновременное тестирование стратегии на нескольких инструментах.
- Общий капитал делится между инструментами согласно правилам управления позицией.
- Метрики отображаются как сводно по портфелю, так и отдельно по каждому инструменту.

### 4.5 Дашборд результатов бэктеста [Must]

Страница `/backtests/:id` с четырьмя вкладками:

**«Обзор» — сводные метрики (сетка 4×2):**

| Метрика | Описание | Цветовая кодировка |
|---|---|---|
| Чистая прибыль | Абсолютная (₽) и в % | >0 зелёный, <0 красный |
| Количество сделок | Всего | Нейтральный |
| Доля прибыльных (Win Rate) | % прибыльных сделок | >55% зелёный, 40-55% нейтральный, <40% красный |
| Профит-фактор | Сумма прибыли / Сумма убытков | >1.5 зелёный, 1.0-1.5 нейтральный, <1.0 красный |
| Макс. просадка | % от капитала | Всегда нейтральный |
| Коэффициент Шарпа | По дневным доходностям | >1.0 зелёный, 0.5-1.0 нейтральный, <0.5 красный |
| Ср. P&L на сделку | Среднее по ВСЕМ сделкам | >0 зелёный, <0 красный |
| Ср. длительность | В часах (внутридневные) или днях | Нейтральный |

Benchmark-сравнение: горизонтальные бары (Стратегия / Buy & Hold / IMOEX).

**«График»:**
- Свечной график инструмента с маркерами сделок: вход ▲ зелёный, выход ▼ красный (размер 2x, текст — лоты, фиксированный шрифт).
- Зоны сделок: цветные полосы между входом и выходом (зелёная = прибыль, красная = убыток).

**«Показатели»:**
- Equity curve (LineSeries зелёная) + drawdown (AreaSeries красная) + Buy & Hold (серая) + IMOEX (синяя).
- Toggle ₽ / % для переключения шкалы.
- Метрики в левой панели (не на графике).

**«Сделки»:**
- Таблица: #, Тип (Long/Short), Вход (дата+цена), Выход (дата+цена), Объём, P&L (зелёный/красный), Длительность, Комиссия.
- Фильтр: Все (N) / Прибыльные (N) / Убыточные (N) — SegmentedControl с счётчиками.
- Сортировка по колонкам, пагинация (10 записей/страница).
- Экспорт в CSV.

Кнопки CSV и PDF в header страницы. Результаты автоматически сохраняются в БД с привязкой к версии стратегии.

### 4.6 Walk-forward анализ [Could]

- Опциональный режим out-of-sample валидации: пользователь задаёт долю in-sample / out-of-sample (например, 70% / 30%).
- Система оптимизирует параметры на in-sample периоде, затем тестирует лучшую комбинацию на out-of-sample.
- Результат: сравнительная таблица in-sample vs out-of-sample метрик с визуальным выделением расхождений.
- Предупреждение: если out-of-sample результаты значительно хуже in-sample — индикатор переобучения (overfitting).

### 4.7 Учёт НКД для облигаций [Must] — ✅ реализовано S5

- При бэктесте на облигациях P&L рассчитывается с учётом накопленного купонного дохода (НКД) на момент покупки и продажи.
- **Bond Service:** отдельный модуль `app/market_data/bond_service.py` — расчёт НКД, парсинг параметров выпуска из MOEX ISS API (дата купона, ставка, номинал, периодичность).
- В боевом режиме: НКД запрашивается через брокерский API при расчёте фактического P&L.
- В интерфейсе: P&L по облигациям отображается с разбивкой: ценовая составляющая + НКД.

## 5. Графический модуль

### 5.1 Тип графика и данные [Must]

- Свечной график OHLCV (candlestick).
- Отображение объёма в нижней панели.
- Выбор таймфрейма непосредственно на графике.

> **Примечание для ТЗ:** Рекомендованная библиотека — TradingView Lightweight Charts (open source, MIT-лицензия). Окончательный выбор определяется в ТЗ.

### 5.2 Подключаемые индикаторы [Must]

- Пользователь добавляет индикаторы из библиотеки через панель.
- Минимальный обязательный набор: SMA, EMA, Bollinger Bands, RSI, MACD, Stochastic, ATR, Volume MA.
- Индикаторы, заданные в стратегии, отображаются на графике автоматически при просмотре результатов.
- Каждый индикатор настраивается: период, цвет, толщина линии.

### 5.3 Инструменты рисования [Should]

- Линии тренда (свободная линия, луч, отрезок).
- Горизонтальные уровни.
- Прямоугольники (зоны).
- Сохранение разметки в БД — привязка к инструменту и таймфрейму.

### 5.4 Мультитаймфрейм [Could]

- Split-view: два графика одного инструмента на разных таймфреймах на одном экране (например, 1ч + Д).
- Синхронизация: вертикальный курсор на одном графике отражается на другом.

### 5.5 Real-time обновление графиков [Must] — ✅ реализовано S5 + S5R + S6

- Графики обновляются в реальном времени **всегда**, когда пользователь открывает график инструмента — независимо от запущенных стратегий. Система создаёт WebSocket-подписку на выбранный инструмент через брокерский API.
- При закрытии графика подписка отменяется для экономии ресурсов.
- **Маркеры сделок на графике [S6]:** трёхуровневый маркер (стрелка + Buy/Sell + цена) для каждой сделки сессии. Загрузка исторических маркеров при mount, реалтайм-обновление через WS. Маркеры хранятся отдельно от свечей (`sessionTradesRef`), пересчитываются после `setData()` и при смене таймфрейма — корректное отображение в sequential mode и при подгрузке старых свечей.
- **Overlay OHLC [S6]:** постоянная панель с OHLC-значениями наведённой свечи (или последней, если курсор вне графика). Для D-таймфрейма фильтрация WS-свечей (S6 chart-stability).
- **Sequential mode:** для intraday (1m/5m/15m/1h/4h) бары идут подряд без ночных разрывов. Для D/W/M — обычный режим. Переключение `Д / 1Ч / 5М / ...` в header графика.
- **Фильтрация аукционных свечей (1h/4h):** свечи с O=H=L=C удаляются в `market_data/service.py` (не отображаются в TradingView).
- **Price alerts lines (S6):** при наличии активных алертов — горизонтальные линии на графике с оранжевыми бейджами и кнопкой удаления.

### 5.6 Стакан заявок (Order Book) [Could]

- Опциональная панель глубины рынка (Level 2): отображение лучших N заявок на покупку и продажу (по умолчанию 10 уровней).
- Данные поступают через брокерский API в реальном времени.
- Отображается в боковой панели справа от графика.
- Визуальное выделение крупных заявок (объём > N × средний объём уровня).

### 5.7 Лента сделок (Time & Sales) [Could]

- Поток последних сделок по инструменту: время, цена, объём, направление (покупка/продажа — определяется по отношению к спреду).
- Цветовое кодирование: покупки — зелёный, продажи — красный.
- Отображается в нижней или боковой панели (настраиваемое расположение).

### 5.8 Ценовые алерты [Should] — ✅ реализовано S6

- Пользователь устанавливает уровень цены на графике (горизонтальная линия с пометкой «Алерт»).
- При пересечении ценой заданного уровня (сверху вниз или снизу вверх — настраивается) — уведомление в Telegram, email (если событие в `EMAIL_ALLOWED_EVENTS`) и in-app через `NotificationService.create_notification()`.
- Алерты сохраняются в БД (`PriceAlert`), привязаны к инструменту.
- `PriceAlertMonitor` подключён к `GET /api/v1/market-data/candles` — при каждом запросе проверяет последнюю свечу против активных алертов (событие `price_alert`).
- Управление алертами: список активных алертов в панели графика справа от кнопки создания, с бейджами-ценами и кнопкой удаления (`IconBellOff`). Сработавший алерт автоматически ставится в `is_active=False`.

## 6. Торговля: режимы «Боевой» и «Бумажный» (Paper Trading) — ✅ реализовано S5 + S6

**Статус реализации:**
- **S5 (2026-04):** Trading Engine (SessionManager, SignalProcessor, OrderManager, PositionTracker), Paper Trading (PaperBrokerAdapter, instant fill), Circuit Breaker (9 проверок), Bond НКД, Tax FIFO + 3-НДФЛ экспорт.
- **S5R (2026-04):** Live Runtime Loop замкнут (`runtime.py:_handle_candle`), реальные позиции T-Invest через FIGI source-match.
- **S6 (2026-04):** Paper+Real recovery при рестарте, Graceful Shutdown (pending orders timeout 30s), Stream Multiplex (один gRPC stream на все подписки), расширенные карточки сессий (P&L + unrealized, W/L, маркеры графика), PauseConfirmModal, Circuit Breaker temporary блокировки (trading_hours, cooldown).

### 6.1 Переключатель режима [Must]

- Каждый запущенный экземпляр стратегии (стратегия + инструмент + брокер) имеет явный переключатель: **Real / Paper**.
- Переключение из Paper → Real требует дополнительного confirm-диалога.
- Режим визуально отмечен во всём интерфейсе: плашка «PAPER» (жёлтый) или «REAL» (красный) на всех связанных экранах.

### 6.1.1 Таймфрейм торговой сессии [Must]

- При запуске торговой сессии (Paper или Real) пользователь **обязан** выбрать таймфрейм работы стратегии. Поддерживаемые значения: `1m`, `5m`, `15m`, `1h`, `4h`, `D`, `W`, `M`. Значение по умолчанию в UI — `D`.
- Стратегия реагирует на **закрытие бара** выбранного таймфрейма, а не на каждую тиковую котировку. Это гарантирует воспроизводимость между бэктестом и live-режимом.
- На каждом закрытии бара стратегия получает **скользящее окно последних N свечей** (N = 200 по умолчанию), а не одну последнюю свечу. Это нужно для индикаторов с lookback (SMA/EMA/RSI/MACD).
- Таймфрейм сохраняется в модели `TradingSession` и отображается в карточке активной сессии. Для legacy-сессий без указанного таймфрейма (созданных до S5R) показывается прочерк.
- Circuit Breaker проверки (дневной убыток, max drawdown, лимит сделок) вычисляются **на закрытии свечи текущего таймфрейма**, не на каждый тик — см. раздел 12.4.

### 6.2 Бумажная торговля (Paper Trading) [Must]

- Система получает котировки в реальном времени от брокера (те же потоки данных, что и для реальной торговли).
- Применяет логику стратегии к живым данным: генерирует сигналы, имитирует исполнение ордеров по рыночной цене с учётом настроенных комиссий и проскальзывания.
- **Виртуальный портфель:** начальный баланс задаётся пользователем вручную. P&L, открытые позиции и статистика считаются виртуально.
- Уведомления в Telegram и email отправляются с явной пометкой `[PAPER]`.
- Результаты виртуальных сделок сохраняются в БД **отдельно** от реальных — для последующего сравнения бумажной и боевой торговли.
- Использование Paper Trading как «прогрев» перед переключением в Real: система может отображать накопленную статистику Paper за текущий период рядом с результатами бэктеста.
- **Восстановление при перезапуске:** Paper-сессии восстанавливаются аналогично Real-сессиям (см. раздел 17.7). Виртуальный баланс и открытые позиции сохраняются в БД. При старте системы — автоматическая подписка на котировки и возобновление мониторинга сигналов. Уведомление пользователю с пометкой `[PAPER]` о восстановлении сессии.

### 6.3 Боевая торговля (Real) [Must]

- **Запуск стратегии:** выбор брокерского аккаунта, конкретного счёта (см. раздел 7.7), инструмента, режима расчёта размера позиции. Первый запуск требует явного подтверждения.
- **Проверка конфликтов при запуске:** система проверяет, нет ли уже активной стратегии на том же инструменте и брокерском счёте (см. раздел 3.5). Если есть — блокировка с предложением остановить существующую стратегию.
- **Ордера:** рыночные, лимитные, стоп-маркет, стоп-лимит ордера на покупку/продажу в соответствии с сигналами стратегии (см. раздел 7.2).
- **Управление позициями:**
  - Принудительное закрытие отдельной позиции из интерфейса.
  - Закрытие всех позиций одной командой (требует подтверждения).
  - Пауза стратегии: сигналы входа блокируются, открытые позиции удерживаются.
- Все сделки (открытие / изменение / закрытие) автоматически логируются в БД с полными атрибутами.
- **Нехватка средств:** если на счёте недостаточно средств для открытия позиции — сигнал пропускается, уведомление в Telegram и in-app с указанием причины. Стратегия продолжает работу. При 3 пропусках подряд из-за нехватки средств — автоматическая пауза стратегии с уведомлением.

### 6.4 Учёт корпоративных действий в боевом режиме [Must]

- **Сплиты:** при сплите акции система автоматически корректирует цену входа и объём открытой позиции для корректного расчёта P&L.
- **Дивиденды:** дивидендные выплаты по акциям учитываются в P&L стратегии (как дополнительный доход). Система отслеживает дивидендные события через MOEX ISS API и брокерский API.
- **Купоны по облигациям:** купонные выплаты учитываются в P&L стратегии. НКД при покупке/продаже учитывается при расчёте финансового результата (см. также раздел 4.7).
- При наступлении корпоративного действия по инструменту с открытой позицией — уведомление пользователю с деталями.

## 7. Интеграция с брокерами

### 7.1 Поддерживаемые брокеры [Must]

**Приоритет фазы 1:** T-Invest (Tinkoff Invest API).

**Архитектура:** абстрактный интерфейс брокера (Broker Adapter / plugin-pattern). Каждый брокер реализует единый интерфейс. Добавление нового брокера не требует изменения ядра системы.

Планируемые брокеры для фазы 2: АЛОР Брокер (OpenAPI), Финам (TRANSAQ / REST), ВТБ, Сбер.

### 7.2 Функции брокерского адаптера [Must]

| Функция | Описание |
|---|---|
| Получение исторических данных | OHLCV за запрошенный период и таймфрейм |
| Рыночные котировки в реальном времени | WebSocket / gRPC stream подписка |
| Поиск инструмента | По тикеру, ISIN, названию |
| Размещение ордеров | Рыночный, лимитный, стоп-маркет, стоп-лимит |
| Отмена ордеров | По идентификатору ордера |
| Состояние ордеров | Статус: выставлен / частично исполнен / исполнен / отменён |
| Получение позиций | Список открытых позиций по счёту |
| Баланс счёта | Денежный остаток (включая заблокированные средства), оценка портфеля |
| История операций | Сделки за период |
| Информация об инструменте | Размер лота, минимальный шаг цены, торговый статус, режим расчётов |
| Стакан заявок | Подписка на глубину рынка (Order Book) |
| Лента сделок | Подписка на поток сделок (Trades stream) |

**Стоп-ордера:** если брокер поддерживает серверные стоп-маркет / стоп-лимит ордера — используются нативные ордера брокера. Если брокер не поддерживает — система эмулирует стоп-ордера: мониторит цену в реальном времени и выставляет рыночный/лимитный ордер при достижении стоп-уровня.

**Шаг цены:** брокерский адаптер возвращает минимальный шаг цены для каждого инструмента. Лимитные и стоп-лимит ордера автоматически округляются до ближайшего допустимого шага цены.

### 7.3 Просмотр баланса и состояния счёта [Must]

**Экран «Счёт»** — отдельный раздел приложения:

- Текущий денежный баланс по каждому подключённому брокерскому счёту (включая разбивку: доступные средства / заблокированные).
- Открытые позиции у брокера: в том числе позиции, открытые вне системы (куплено вручную в приложении брокера).
- Нереализованный P&L по открытым позициям.
- История операций за выбранный период (список сделок от брокера).

**Разделение позиций на `strategy` / `external` [Must, S5R]:**
- Каждая позиция в списке имеет поле `source`:
  - `strategy` — открыта одной из активных торговых сессий системы на данном брокерском счёте;
  - `external` — открыта вне системы (вручную в приложении брокера или унаследована).
- В UI поле `source` отображается как badge («Стратегия» / «Внешняя») в колонке `PositionsTable`.
- Определение `source` на backend: match по `ticker + broker_account_id` через таблицу `trading_sessions`/`live_trades` (S5R). В Sprint 6 будет уточнено по FIGI.

**Мульти-валютность балансов и позиций [Must, S5R]:**
- Поддерживаются валюты `RUB`, `USD`, `EUR`, `HKD`, `CNY` — **без автоматической конвертации** в рубли.
- Балансы отображаются раздельно по каждой валюте. Позиции — с указанием валюты инструмента.
- P&L рассчитывается в валюте инструмента; агрегированный рублёвый эквивалент в фазе 1 не вычисляется.

**История операций [Must, S5R]:**
- Типы операций: `BUY`, `SELL`, `DIVIDEND`, `COUPON`, `COMMISSION`, `TAX`, `OTHER`.
- Состояния: `EXECUTED`, `CANCELED`, `PROGRESS`.
- Фильтры: диапазон дат `from` / `to` (обязательные), пагинация `offset` / `limit` (по умолчанию 100, максимум 500).
- UI: колонки тип, дата, тикер, количество, цена, сумма, валюта. Иконки для типов.

**Виджет баланса на главном дашборде:**
- Компактное отображение: доступные средства + суммарный P&L открытых позиций управляемых стратегий.
- Рядом с реальным балансом — виртуальный баланс активных Paper-стратегий с визуальным разделением.

### 7.4 Режим расчётов T+1 [Must]

- Система учитывает режим расчётов T+1 для акций на фондовом рынке MOEX: после покупки средства блокируются до следующего торгового дня.
- При расчёте размера позиции учитываются только фактически доступные (незаблокированные) средства — информация запрашивается через брокерский API.
- В бэктесте: опциональный учёт T+1 (по умолчанию включён для акций). При включении — после покупки виртуальные средства блокируются на 1 торговый день.

### 7.5 Частичное исполнение ордеров [Must]

- При частичном исполнении лимитного ордера: система ожидает полного исполнения в течение настраиваемого таймаута (по умолчанию 60 секунд).
- По истечении таймаута: неисполненный остаток автоматически отменяется, позиция фиксируется по фактически исполненному объёму.
- Уведомление пользователю о частичном исполнении (in-app и Telegram) с указанием исполненного и отменённого объёмов.
- Стратегия продолжает работу с учётом фактического объёма позиции.

### 7.6 Фиксация проскальзывания [Must]

- В боевом режиме: для каждой сделки фиксируется фактическое проскальзывание — разница между ценой сигнала стратегии и фактической ценой исполнения ордера у брокера.
- Статистика проскальзывания отображается в отчёте по стратегии: среднее, максимальное, суммарное за период.
- Рекомендация пользователю: если фактическое проскальзывание систематически превышает настроенное в бэктесте — предупреждение с предложением скорректировать параметры.

### 7.7 Несколько счетов одного брокера [Must]

- T-Invest (и другие брокеры) позволяют иметь несколько счетов: брокерский, ИИС и др.
- При подключении API-ключа система запрашивает список доступных счетов через брокерский API и предлагает пользователю выбрать один или несколько для подключения.
- Каждый счёт добавляется как отдельная запись с указанием типа счёта (брокерский / ИИС).
- При запуске стратегии — обязательный явный выбор конкретного счёта из списка подключённых.

### 7.8 Работа с rate limits брокерского API [Must]

- Брокерский адаптер учитывает ограничения API провайдера (для T-Invest: 300 unary-запросов в минуту, лимиты на объём исторических данных за один запрос).
- Очередь запросов с автоматическим throttling: запросы распределяются равномерно в пределах лимита.
- При приближении к лимиту — замедление запросов (exponential backoff). При превышении (HTTP 429) — автоматическая пауза и повтор после указанного периода.
- Загрузка исторических данных выполняется порциями в пределах ограничений API (для T-Invest: не более 1 года данных за один запрос для дневного таймфрейма).

## 8. AI-ассистент

### 8.1 Роль AI в системе [Must]

AI-ассистент выполняет следующие функции:
1. Помогает пользователю грамотно сформулировать торговую стратегию в тексте (задаёт уточняющие вопросы, исправляет терминологию).
2. Преобразует текстовое описание в структурированный набор блоков (правила входа, выхода, индикаторы).
3. Генерирует Python-код для Backtrader по блочной схеме.
4. По запросу объясняет, что делает тот или иной блок или сгенерированный код.

### 8.2 Интерфейс AI — два режима [Must]

**Режим A — Встроенный AI-ассистент (online):**
- Sidebar-чат на странице создания/редактирования стратегии.
- Пользователь описывает стратегию → AI предлагает блочную схему → пользователь корректирует → AI пересобирает код.
- История диалога сохраняется вместе со стратегией.
- **При недоступности LLM-модели:** в sidebar-чате отображается предупреждение: «AI-ассистент недоступен. Проверьте настройки API-ключа и доступность провайдера. Вы можете создать стратегию вручную в блочном редакторе или вставить описание из внешнего источника (Режим B)».

**Режим B — Внешний AI / ручной ввод:**
- Пользователь может подготовить описание стратегии во внешнем AI-инструменте (ChatGPT, Claude и т.д.) и вставить результат в систему.
- Кнопка **«Скопировать шаблон»** — копирует в буфер обмена стандартный шаблон описания стратегии (см. Приложение A).
- Поле **«Вставить описание из внешнего источника»** — система парсит вставленный текст по шаблону и конвертирует в блочное представление **без обращения к AI**.
- Если формат не распознан — сообщение с деталями ошибки и предложение доработать описание по шаблону.
- Кнопка «Открыть шаблон в новой вкладке» — полная инструкция по заполнению шаблона с примерами.

### 8.3 Требования к AI-интеграции [Must]

- **Pluggable providers:** подключение через единый интерфейс — смена модели через настройки без изменения кода. Поддерживаемые типы: Anthropic Claude, OpenAI, DeepSeek, Google Gemini, Mistral AI, Groq, Alibaba Qwen, OpenRouter, Custom (OpenAI-совместимый API).
- **Хранение ключей:** API-ключи AI-провайдеров хранятся в БД (таблица `ai_provider_configs`), зашифрованные AES-256-GCM. Fallback: если в БД нет активного провайдера — используются переменные `.env`.
- **Настройки AI:** вкладка «AI-провайдеры» в разделе Настройки (`/settings?tab=ai`). Таблица провайдеров с маскированными ключами (`sk-ab...xyz1`), кнопка верификации ключа без сохранения, radio-выбор активного провайдера.
- **Множество провайдеров:** пользователь может добавить несколько провайдеров и переключаться между ними. Dropdown выбора модели в header AI-чата.
- Генерируемый код проходит обязательную валидацию перед исполнением (см. раздел 12.5).
- **Контекст:** системный промпт включает справку о MOEX (торговые часы, лотность, рублёвые инструменты, доступные индикаторы, ограничения). AI проверяет каждый индикатор и условие на реализуемость в Blockly.
- **Ограничение контекста:** при превышении лимита контекстного окна — автоматическое сжатие истории (Context Compressor). Предупреждение: «История диалога сжата для продолжения работы».
- **Rate limits и бюджет:** настраиваемые лимиты запросов (в сутки и в месяц) per-провайдер. Значение 0 = без ограничений (∞). Отображение счётчика и прогресс-баров в настройках. При исчерпании — автоматический переход в ручной режим.
- **SSE Streaming:** ответы AI доставляются через Server-Sent Events (SSE) с потоковым отображением текста. Поддержка concurrent streams (MAX_CONCURRENT_STREAMS = 3).
- **Fallback при недоступности AI:** предупреждение в AI-режиме + ссылка «Перейти в настройки». Блочный редактор доступен **всегда**.

> **Примечание для ТЗ:** Конкретные алгоритмы хеширования (bcrypt / Argon2id), параметры JWT, TTL токенов определяются в ТЗ.

### 12.2 Защита API-ключей брокеров

- Хранение: шифрование в БД. Ключ шифрования — только в env-переменной, никогда не записывается в БД.
- Принцип минимальных прав: пользователю рекомендуется создавать API-ключи брокера **только с торговыми правами**, без прав вывода денежных средств. Система отображает предупреждение, если ключ имеет права вывода (детектируется при первом подключении).
- В любых логах, ошибках и UI — ключи отображаются только в маскированном виде: первые 4 и последние 4 символа.
- Из интерфейса: кнопка «Отозвать ключ» немедленно помечает аккаунт как неактивный и останавливает все связанные стратегии.

> **Примечание для ТЗ:** Конкретный алгоритм шифрования (AES-256-GCM / аналог), процедура ротации ключей определяются в ТЗ.

### 12.3 Защита Telegram-бота

- Бот обрабатывает только сообщения от зарегистрированного chat_id. Все остальные запросы игнорируются без ответа.
- Опасные команды (`/closeall`, `/close`) требуют inline-подтверждения с кнопками «Подтвердить» / «Отмена» с TTL 60 секунд.
- Webhook-запросы Telegram верифицируются по секретному токену.

### 12.4 Защита торговых операций (Circuit Breaker) [Must] — ✅ реализовано S5 + S6 fixes

> **Реализация:** `CircuitBreakerEngine` (9 проверок), per-user `asyncio.Lock` (Gotcha 5), записывает `CircuitBreakerEvent`. Вызывается из `runtime.py:_handle_candle` → `SignalProcessor.process_candle` → `check_before_order`. Все проверки — на закрытии свечи таймфрейма сессии, не на каждый тик (воспроизводимость между бэктестом и live).

**Постоянные блокировки (требуют ручного возобновления):**
- **Дневной лимит убытков:** настраивается в профиле пользователя (раздел 2.2). При превышении: все стратегии переводятся в паузу, уведомление отправляется во все каналы. Возобновление — только вручную.
- **Максимальная просадка (max drawdown):** настраивается в профиле (% от пикового баланса). При достижении — все стратегии на паузу, уведомление, возобновление только вручную.
- **Лимит размера позиции:** максимальная сумма одного ордера (настраивается в профиле). Ордера сверх лимита блокируются с уведомлением.
- **Лимит сделок в день:** настраиваемый порог (по всем стратегиям суммарно). При превышении — блокировка новых сигналов входа до следующего торгового дня с уведомлением.
- **Запрет одновременной разнонаправленной торговли:** система запрещает открытие позиции в противоположном направлении по тому же инструменту на том же брокерском счёте, пока текущая позиция не закрыта. В фазе 1 Short-ордера блокируются на уровне валидации (архитектура готова к активации в фазе 2). Попытка блокируется с уведомлением.
- **3+ последовательных пропуска из-за нехватки средств:** автоматическая пауза стратегии с уведомлением (раздел 6.3).

**Временные блокировки (автовозобновление, S6 fix):** `CheckResult.temporary=True` — стратегия остаётся `active`, CB-событие не записывается, уведомление не шлётся:
- **Trading hours 10:00–23:50 MSK** (MOEX основная + вечерняя сессия): сигналы вне окна пропускаются молча, утром/после открытия возобновляются без ручного вмешательства.
- **Cooldown-период:** минимальный интервал между двумя сигналами входа одной стратегии (настраивается в параметрах стратегии). `last_signal_at` обновляется **после** успешного создания сделки в `runtime.py`, не в `SignalProcessor` — чтобы cooldown не блокировал повторные попытки при неудачных ордерах.

**Прочее:**
- **Подтверждение первого запуска:** переход стратегии в режим Real требует явного confirm-диалога с отображением параметров (инструмент, размер позиции, брокер, счёт).
- **PauseConfirmModal (S6):** при ручной паузе сессии с открытой позицией — диалог с вариантами «Отмена», «Пауза, позицию оставить», «Закрыть позицию и пауза». Если позиций нет — пауза выполняется сразу.
- **Режим «заморозки» при потере связи:** см. раздел 14.2.

### 12.5 Изоляция генерируемого кода AI → Backtrader [Must]

- Сгенерированный Python-код выполняется в изолированной среде с использованием RestrictedPython.
- **AST-анализ перед исполнением:** блокируются любые импорты вне белого списка (`backtrader`, `pandas`, `numpy`, `ta`, `datetime`), файловые операции, сетевые вызовы, системные команды.
- **Блокировка escape-техник:** AST-анализ дополнительно запрещает: `eval`, `exec`, `compile`, `__import__`, `getattr`, `setattr`, `globals`, `locals`, `type.__subclasses__`, `__builtins__`.
- **Ограничение ресурсов:** лимит CPU (30 секунд) и RAM (512 МБ). Превышение лимитов — принудительное завершение процесса.
- При обнаружении запрещённых конструкций — код не запускается, пользователю показывается сообщение об ошибке с деталями.

> **Примечание для ТЗ:** Конкретный механизм изоляции (RestrictedPython + subprocess / альтернативы) и способ ограничения ресурсов определяются в ТЗ.

### 12.6 Транспортная безопасность и CSRF

- HTTP допускается для локального доступа (127.0.0.1 / localhost). HTTPS обязателен при сетевом доступе (фаза 2).
- WebSocket соединения — WSS при сетевом доступе.
- CSRF-токены обязательны для всех форм, инициирующих торговые действия (запуск стратегии, отмена ордера, закрытие позиций).

> **Примечание для ТЗ:** Заголовки безопасности (CSP, X-Frame-Options и др.) определяются в ТЗ.

### 12.7 Аудит-лог [Must]

- Таблица аудит-лога — append-only (только INSERT).
- Фиксируются события: вход/выход из системы, изменение пароля/ключей, запуск/остановка стратегии, каждый ордер (включая ручные закрытия и команды из Telegram), срабатывание circuit breaker, восстановление после перезапуска.
- Каждая запись включает: время, пользователь, действие, сущность, детали, источник (web UI / Telegram / scheduled / system).

### 12.8 Rate limiting внутреннего API

- Защита от flood-запросов: лимит на количество API-вызовов в минуту (настраиваемый, по умолчанию 100 req/min).
- Отдельный лимит для торговых операций (ордера): 10 req/min.
- При превышении — ответ с кодом «слишком много запросов» и указанием времени ожидания.

### 12.9 Защита env-переменных

- `.env` файл: права доступа ограничены владельцем. Система проверяет права при запуске и предупреждает при небезопасных настройках.
- `.env` обязательно включён в `.gitignore`.
- Шаблон `.env.example` без реальных значений поставляется с приложением.
- При запуске: проверка наличия обязательных переменных. При отсутствии — блокировка старта с понятным сообщением.

### 14.1 Экспорт данных [Should]

### 14.2 Устойчивость к сбоям соединения с брокером [Must]

- Автоматическое переподключение: exponential backoff (1с → 2с → 4с → 8с → … → 60с, максимум 10 попыток).
- При потере связи во время открытой позиции — уведомление во все каналы с указанием последнего известного состояния и отметкой времени обрыва.
- **Режим «заморозки»:** новые сигналы входа блокируются до восстановления соединения и явного подтверждения пользователем.
- После восстановления соединения: сверка состояния позиций у брокера с состоянием в БД — оповещение о расхождениях.

### 14.3 Обработка ошибок исполнения ордеров [Must]

- Если ордер отклонён брокером (нет ликвидности, недостаток средств, ограничения по инструменту) — уведомление в Telegram и in-app с кодом и описанием ошибки.
- Стратегия не останавливается автоматически при единичной ошибке, но логирует её. После N ошибок подряд (настраивается, по умолчанию 3) — автоматическая пауза с уведомлением.

### 14.4 Требования к тестированию

- **Unit-тесты:** расчёт P&L (с учётом комиссий, лотности, НКД для облигаций), расчёт метрик бэктеста, логика circuit breaker (все виды лимитов), валидация OHLCV-данных.
- **Интеграционные тесты:** брокерский адаптер (mock-сервер, эмулирующий T-Invest API), валидация sandbox (попытки escape), парсинг шаблона стратегии (режим B).
- **E2E-тесты:** полный цикл — создание стратегии → бэктест → paper trading → проверка результатов. Сценарии circuit breaker. Сценарии восстановления после перезапуска.

## 16. Ограничения и допущения первой версии

1. **Один пользователь, локальная машина** — авторизационный модуль проектируется с расчётом на мультипользовательский режим (фаза 2), но текущая реализация обслуживает одного.
2. **Нет мобильной версии** — адаптивная вёрстка под планшет (≥ 1024px) желательна, мобильная (< 768px) — вне scope.
3. **Нет публичного REST API** — внешние интеграции не поддерживаются в фазе 1.
4. **Только длинные позиции (long)** в фазе 1 — Short-ордера блокируются на уровне валидации, но архитектура проектируется с учётом активации Short и маржинальной торговли в фазе 2.
5. **Срочный рынок FORTS (фьючерсы)** — не поддерживается в фазе 1. Архитектура брокерского адаптера предусматривает расширение на фьючерсы в фазе 2 (включая учёт ГО, вариационной маржи, клирингов).
6. **Опции и структурные продукты** — не поддерживаются.
7. **Автоматический перенос позиций через ночь** — учёт гэпов при открытии торгов обрабатывается Backtrader стандартными методами; специальной логики не требуется в фазе 1.
8. **Accessibility (WCAG)** — не входит в scope фазы 1.
9. **Интернационализация (i18n)** — интерфейс только на русском языке. Архитектура i18n не требуется в фазе 1.

### 17.7 Восстановление после перезапуска [Must] — ✅ реализовано S5R + S6

### 19.1 Версионирование стратегий [Should] — ✅ S7.7.1

- При каждом сохранении стратегии backend автоматически создаёт новую запись `strategy_versions` (snapshot полного `blocks_json` + `generated_code`).

### 19.2 Grid Search оптимизация параметров [Should] — ✅ S7.7.2

- Запуск из редактора стратегии: «Grid Search» открывает модалку с динамическим списком параметров (1–5).
- Real-time счётчик `total = product(len(values))` с цветовой индикацией: ≤200 зелёный, 201–1000 жёлтый, >1000 красный (запуск заблокирован).
- Hard cap: 1000 комбинаций / ≤5 параметров (защита от 10⁶).
- Backend выполняет каждую комбинацию через `multiprocessing.Pool` (workers = `os.cpu_count() - 1`), pickle-friendly worker.
- Результат — матрица `[{params, sharpe, pnl, win_rate, drawdown, trades_count}]`, сортировка по убыванию Sharpe.
- **Overfitting-предупреждение:** если разброс топ-5 по Sharpe > 50% — флаг в финальном payload.
- Визуализация в UI: bar (1 параметр) / 2D heatmap (2 параметра) / sortable table (3+ параметра).
- Результаты доступны через тот же канал, что фоновые бэктесты (см. §19.10).

### 19.3 Экспорт CSV/PDF бэктеста [Should] — ✅ S7.7.3

- На странице результата бэктеста — кнопки «Экспорт CSV» и «Экспорт PDF».
- Endpoint `GET /api/v1/backtest/{id}/export?format=csv|pdf`.

### 19.7 Backup/restore [Should] — ✅ S7.7.9

- **Автоматический backup:** APScheduler-job `backup_db_daily` запускается в 03:00 UTC (настраивается через `settings.BACKUP_CRON`).
- **Ротация:** хранятся последние N (по умолчанию 7) snapshot'ов; старые удаляются по mtime.
- **CLI:** `python -m app.cli.backup {create,list,restore,rotate}` — argparse-интерфейс.
- **SQLite WAL-aware copy:** `BEGIN IMMEDIATE` → `PRAGMA wal_checkpoint(FULL)` → `copy2` + копирование `*.db-wal`/`*.db-shm` (см. gotcha-19).
- **Postgres-ветка:** `pg_dump --no-owner --format=custom` → `pg_restore --single-transaction` (готова, активируется по схеме URL).
- После restore автоматически запускается `alembic upgrade head` (схема может быть старее).

- Frontend: `setInterval(fetchSessions, 10000)` polling **полностью удалён**, заменён на WS-клиент с exponential backoff reconnect и ping/pong.

### 19.12 Пять новых event_type [Should] — ✅ S7.7.13 (MR.5)

  - `partial_fill` — частичное исполнение ордера (real-mode, при `filled_lots < volume_lots`).
  - `order_error` — ошибка выставления ордера (helper `_publish_order_error`, ловит exceptions в `process_signal` и `on_order_filled`).

  - `connection_lost` / `connection_restored` — потеря/восстановление T-Invest gRPC-стрима (с анти-спам флагом — один publish на disconnect-цикл).
- Все 5 типов идут в EVENT_MAP, доставляются через 3 канала (in-app + Telegram + Email) согласно настройкам пользователя.
- NotificationService подписан на новый канал `broker:status` (фан-аут по активным сессиям через `_session_user_map`).
