# ⚠️ Integration Verification Checklist

> **Ключевая секция, без которой промпт не валиден.** Без этого чеклиста DEV-агент реализует класс, но **не** замкнёт runtime-цикл. В Sprint 5 это привело к тому, что `SignalProcessor.process_candle()` был реализован и формально принят ARCH-ревью, но в production-коде нигде не вызывался — оформлено как **техдолг #4 High → задача 6.0 Live Runtime Loop в S6**.

Для **каждого нового класса / метода / endpoint**, реализованного в этой задаче, выполни:

- [ ] **Grep на инстанциирование:**
  `grep -rn "<ClassName>(" app/` — класс создаётся где-то **кроме** собственного определения и тестов?

- [ ] **Grep на вызов метода:**
  `grep -rn "\.<method_name>(" app/` — метод вызывается где-то в **production-коде**, не только в тестах?

- [ ] **Listener на event_bus / stream_manager:**
  Если новый класс зависит от подписки на event_bus канал (`market:{ticker}:{tf}`, `trades:{session_id}`, `notifications`) — **явно покажи** в отчёте:
  - Где создаётся listener (`event_bus.subscribe(...)`)
  - Кто вызывает регистрацию listener'а (какой endpoint / lifecycle hook / background task)
  - Как listener переживает перезапуск backend (`restore_sessions` или lifespan hook в `main.py`)

- [ ] **Background task / scheduler:**
  Если новый класс требует периодического запуска (scheduled job) — покажи, где регистрируется job в `app/scheduler/service.py` или `main.py` lifespan.

- [ ] **API endpoint (если применимо):**
  Для каждого нового endpoint — покажи, что он:
  1. Зарегистрирован в `main.py` через `app.include_router(...)`
  2. Имеет `Depends(get_current_user)` для авторизации
  3. Защищён CSRF (для POST/PATCH/PUT/DELETE)
  4. Покрыт как минимум одним интеграционным тестом через `TestClient`

- [ ] **Frontend → Backend:**
  Если реализуешь frontend-компонент, который вызывает новый backend-endpoint — подтверди, что endpoint уже реализован (по `Cross-DEV contracts`). Если нет — это блокер, не делай заглушку, верни `БЛОКЕР`.

- [ ] **Если точка вызова не найдена** — интеграция считается **НЕ завершённой**.
      Либо добавь вызов в production-код, либо **явно помети** в отчёте:
      > `⚠️ Runtime integration: NOT CONNECTED — <class/method> не вызывается в runtime. Требуется следующая задача: <X.Y>`.
      Такая пометка **блокирует** формальную приёмку класса ARCH-ревью.

# ⚠️ Формат отчёта (МАНДАТНЫЙ)

> **Ключевая секция.** Оркестратору нужна предсказуемая структура, чтобы быстро понять, что сделано, что сломалось, какие новые контракты доступны другим DEV.

Верни в финальном ответе **только** структурированную сводку **до 400 слов**. Соблюдай формат строго.

**НЕ возвращай:**
- Полный код реализованных файлов (их видно через `git diff`)
- Полный лог tool-вызовов (то, что ты читал через Read / Grep)
- Подробные объяснения, почему ты принял каждое решение

**Верни:**

```markdown

- <bullet list — 5-10 пунктов, крупными мазками>

- **Изменённые:** `<путь>`, `<путь>`
```
