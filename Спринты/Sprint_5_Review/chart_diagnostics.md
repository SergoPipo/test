# Диагностика графиков — S5R

**Дата старта:** 2026-04-15
**Контекст:** Sprint 5 Review (closeout). Спринт 6 ещё не запущен, все изменения и коммиты по диагностике/фиксу идут в рамках S5R.

## Исходный симптом

Пользователь открыл график SBER из торговой сессии #11 «Тестовая» на таймфрейме 1h и сделал два скриншота с разницей в несколько секунд:

- **Скриншот А:** последняя свеча помечена как `15 апр. '26 00:00`
- **Скриншот Б:** последняя свеча помечена как `15 апр. '26 03:00`

Разница между последними свечами — ровно **3 часа**, что совпадает со смещением UTC↔MSK. Обе временные метки приходятся на ночное время MSK, когда MOEX по SBER **не торгует** (основная сессия 10:00–18:45 MSK, вечерняя 19:00–23:50 MSK).

Пользователь также отмечает, что «до сих пор у нас с графиками какая-то беда, на них не отображаются актуальные свечи и не работает то, как у нас было запланировано».

## Рабочие гипотезы

### H1 — UTC vs MSK alignment в агрегации 1h
Один источник (T-Invest / MOEX ISS / cache) выравнивает bucket'ы 1h-свечей по UTC-границам, другой — по MSK. Смена источника между запросами приводит к сдвигу последнего bucket'а на 3 часа.

**Признаки:** ровно 3 часа сдвиг, обе метки кратны часу, наличие трёх источников в `marketDataStore.dataSource`.

### H2 — WS live stream нормализует timestamp иначе, чем исторические
`upsertLiveCandle` в [marketDataStore.ts:214-232](Develop/frontend/src/stores/marketDataStore.ts#L214-L232) сравнивает `candle.timestamp` со значением в store без нормализации. Если backend шлёт live-свечи в UTC, а исторические в MSK (или наоборот), появляется «лишняя» ночная свеча, которая не совпадает ни с одним реальным bucket'ом.

### H3 — Гонка auto-TF и WS-переподписки
При открытии графика из SessionCard срабатывает `setTimeframe` → `fetchCandles` ([ChartPage.tsx:50-58](Develop/frontend/src/pages/ChartPage.tsx#L50-L58)). Одновременно `useEffect` с зависимостью `[ticker, currentTimeframe]` пересоздаёт WS-канал ([ChartPage.tsx:78-94](Develop/frontend/src/pages/ChartPage.tsx#L78-L94)). Возможна ситуация: исторические свечи пришли от одного источника (кэш), а первая live-свеча перезаписала последнюю bucket'у по другой формуле.

### H4 — Нерыночное время попадает в выдачу
MOEX по акциям ночью не торгует, но `00:00` и `03:00` по MSK 15.04.2026 показаны на графике как реальные свечи. Значит, либо backend агрегирует с «пустыми» bucket'ами, либо источник (T-Invest) возвращает сделки с неторговых часов (премаркет? внебиржа?), либо timestamp интерпретируется неверно.

## Находки Фазы 1

### F1 — ROOT CAUSE: MOEX ISS возвращает `begin` в UTC, парсер считает MSK

**Код-источник ошибки:** [backend/app/broker/moex_iss/parser.py:37-40](Develop/backend/app/broker/moex_iss/parser.py#L37-L40)
```python
elif col in ("begin", "end") and isinstance(val, str):
    # MOEX ISS возвращает время в МСК — конвертируем в UTC для единообразия с T-Invest
    msk_dt = datetime.strptime(val, "%Y-%m-%d %H:%M:%S").replace(tzinfo=MOSCOW_TZ)
    record[col] = msk_dt.astimezone(timezone.utc)
```

**Факт:** комментарий и логика неверны. ISS на самом деле возвращает время в **UTC**. Доказательства (запросы к `iss.moex.com/iss/.../SBER/candles.json` от 2026-04-15):

1. **1h-свечи за 2026-04-14, взятые как plain ISS-строки (якобы «MSK»):**
   - `06:00` volume=1537 — мелкий объём
   - `07:00` volume=266281 — резкий рост
   - `08:00–19:00` — пиковый дневной torgующий ряд
   - `20:00–23:00` volume=114K–292K — средний спад (extended)

   Если это MSK: `06:00 MSK` и `07:00 MSK` — до начала основной сессии (10:00 MSK), volume должен быть ≈0. Противоречие.

   Если это UTC: `06:00 UTC = 09:00 MSK` (утро перед открытием, малый volume ✓) и `07:00 UTC = 10:00 MSK` (ровно открытие основной сессии, резкий рост ✓). **Полное согласование.**

2. **1m-свечи за 2026-04-14 06:55–07:05 (ISS):**
   ```
   2026-04-14 06:59:00  volume=1537
   2026-04-14 07:00:00  volume=10726
   2026-04-14 07:01:00  volume=6981
   ...
   ```
   `06:59` с volume=1537 — это точно **09:59 MSK**, премаркет-аукцион перед 10:00 MSK open. Эта секунда невозможна в MSK-интерпретации.

**Заключение:** ISS API возвращает timestamps в UTC. Парсер ошибочно вешает на них MSK tz и делает `.astimezone(UTC)` → сдвиг **−3h**. Все свечи из ISS хранятся в кэше со сдвигом −3h относительно правильного UTC.

### F2 — Frontend +3h compensation частично маскирует ошибку

**Код:** [frontend/src/components/charts/CandlestickChart.tsx:26-33](Develop/frontend/src/components/charts/CandlestickChart.tsx#L26-L33)
```ts
function parseUtcTimestamp(ts: string): number {
  const utcStr = ts.endsWith('Z') || ts.includes('+') ? ts : ts + 'Z';
  return Math.floor(new Date(utcStr).getTime() / 1000) + MSK_OFFSET_SEC;
}
```
Frontend добавляет `+3h`, чтобы lightweight-charts (всегда UTC axis) визуально показывал MSK. Но этот трюк предполагает, что в API приходит **корректный UTC**. Когда ISS-парсер прислал свечу со сдвигом −3h (фактический unix = 10:00 MSK − 6h = 04:00 UTC вместо 07:00 UTC), frontend делает +3h → отображает как **07:00** на оси вместо правильного **10:00**.

**Результат:** график SBER на 1h стабильно сдвинут на −3 часа от реального MSK. Основная сессия показывается как 07:00–15:00 вместо 10:00–18:00, вечерняя — как 16:00–20:00 вместо 19:00–23:00.

### F3 — «Ночные фантомы» на скриншотах пользователя

Скриншоты показывают последние свечи `15 апр. '26 00:00` и `15 апр. '26 03:00`. Обратная цепочка:

- Chart label `"00:00"` → unix = seconds(2026-04-15T00:00Z) → backend naive UTC `2026-04-14 21:00` (после вычета +3h frontend offset) → ISS raw string был `"2026-04-15 00:00:00"` → фактически это 00:00 UTC = **03:00 MSK** (extended hours после закрытия вечерней сессии 23:50 MSK).
- Chart label `"03:00"` → ISS raw `"2026-04-15 03:00:00"` → фактически 03:00 UTC = **06:00 MSK** (утренний малоликвидный премаркет).

Обе свечи — это реальные extended hours / premarket данные MOEX, но подписаны на графике на 3 часа раньше реального времени. Это не «фантомы», это нерыночные extended-часы, неправильно смещённые.

**Разница между двумя скриншотами 3 часа** — не магия. Между запросами ISS добавила в ответ одну новую свечу (или `_build_current_candle` продвинулся на час), а из-за −3h сдвига разница между «последней свечой» в двух ответах выглядит ровно как 3 часа.

### F4 — Влияние на другие таймфреймы

Баг затрагивает **все свечи ISS-источника**, а не только 1h:
- `5m`, `15m`, `1h`, `4h` — сдвиг на −3h
- `D` (дневные) — ISS возвращает `begin` = `2026-04-15 00:00:00`, парсер считает это MSK → UTC 21:00 14.04. Frontend +3h → 00:00 UTC 15.04 → ось показывает «15 апр». Формально день верный (повезло), но внутренний timestamp — 21:00 14.04 UTC, что некорректно для совпадения с другими источниками.
- `W`, `M` — аналогично D, ошибка «почти не заметна».

`_build_current_candle` использует `datetime.now(timezone.utc)` и настоящий UTC. Его свечи **консистентны с правильным UTC**, но НЕсовместимы с ISS-кэшем (сдвиг 3h). При merge через `by_ts = {c.timestamp: c for c in result}` получаем **дубли одного и того же часа** под разными ключами — это объясняет периодически появляющиеся «лишние» свечи, визуальные глитчи и странные промежутки.

### F5 — Cache накопил повреждённые данные

Таблица `ohlcv_cache` содержит свечи со сдвигом −3h, пришедшие из ISS за всё время работы backend. Даже после фикса парсера нужно **очистить/перезалить** ISS-часть кэша, иначе старые записи продолжат отдаваться неправильно.

## Принятое архитектурное решение (2026-04-15)

См. подробно в [decision_market_data_source.md](decision_market_data_source.md). Кратко:

**T-Invest — единственный источник для viewer/backtest/trading, если он подключён.** MOEX ISS → только viewer-fallback и только при отсутствии T-Invest. Backtest и trading без T-Invest — жёстко блокируются с HTTP 409 и UI-модалкой. Fallback на ISS при пустом ответе T-Invest (вариант B) — **не делаем**, принят **вариант A** (строгий).

Это меняет масштаб фикса: не просто исправить парсер и почистить кэш, а перестроить `MarketDataService.get_candles` с mode-guard и убрать merge-логику `tinvest+iss`.

## Root Cause и план фикса

**Root cause:** одна строка в `parser.py:39-40` (MSK вместо UTC) + накопленный повреждённый кэш.

**Фикс (7 фаз, см. `decision_market_data_source.md` для деталей):**

3. **Фаза 4** — Alembic миграция `DELETE FROM ohlcv_cache` (полная очистка, не только ISS)
4. **Фаза 5** — Frontend: 409-interceptor + `TInvestRequiredModal` + viewer-badge «Данные MOEX»
5. **Фаза 6** — Очистка кода (убрать `tinvest+iss` merge)
6. **Фаза 7** — E2E: Playwright по матрице режимов × тикеров × таймфреймов

**Дополнительно к проверке:**
- T-Invest adapter (`mapper.py:177-181`) — возвращает tz-aware UTC из gRPC. Это корректно, timestamp НЕ сдвигается. ✓
- `_build_current_candle` (`service.py:263`) — использует `datetime.now(timezone.utc)`. Корректно. ✓
- `_raw_to_candle` (`service.py:472-481`) — strip tz к naive UTC. Корректно, при условии что входящий tz правильный. ✓
- Frontend `parseUtcTimestamp` — +3h offset, сам по себе корректен **при условии правильного backend UTC**. После фикса parser'а работает правильно. ✓

**План фикса (Фазы 2–4):**
