# Внеплановое ревью после Sprint 5

> **Статус:** 🔄 в процессе — планирование
> **Открыто:** 2026-04-14
> **Причина:** ретроспектива S5 выявила блокирующий долг, который не даёт корректно начать Sprint 6
> **Охватывает:** Sprint 5 (Торговля / M3 Paper Trading) + накопленный долг S1-S5

## Отличие от плановых ревью

Это **внеплановое промежуточное ревью**. В отличие от `Sprint_2_Review` (M1) и `Sprint_4_Review` (M2), которые были запланированы после каждых 2-х спринтов, `Sprint_5_Review` проводится **экстренно** между S5 и S6. Причина — накопленный технический и процессный долг, который блокирует запуск Sprint 6.

Плановое `Sprint_6_Review` (после S5+S6, M3) **остаётся на месте** и будет проведено после завершения S6. Это ревью — его промежуточный младший брат.

## Почему созвано

По итогам ретроспективы Sprint 5 (2026-04-14, см. `Sprint_5/changelog.md`) выявлены **5 характерных слабостей** промпт-файлов `prompt_DEV-N.md`, которые привели к 41 замечанию ручного тестирования и двум High/Medium техдолгам. Параллельно при подготовке S6 обнаружились ещё два критических факта:

1. **CI красный уже 11 дней** (с 2026-04-03) — backend падает на `ruff check` (ложные F401 в `alembic/env.py`), frontend на `pnpm lint` (React 19 + новые правила `eslint-plugin-react-hooks@7.x`). Из-за этого `mypy`, `pytest`, `tsc --noEmit`, `vitest` вообще не доходят до выполнения (bash останавливается на первой ошибке). Никто не смотрел на письма от GitHub — уведомления копились неделю.

2. **Live Runtime Loop не замкнут** — `SignalProcessor.process_candle()` в коде S5 есть, ARCH-ревью формально принял его ✅, но в production-коде метод **нигде не вызывается**. Цикл «стрим свечей → стратегия → сигнал → ордер» физически разорван. У `TradingSession` нет поля `timeframe`, `scheduler/service.py` пуст. Архитектура в `technical_specification.md:1396-1417` описана, но не реализована.

Кроме того:
- 30+ E2E тестов S4 (AI Chat, Backtest, Blockly) падают с момента Sprint 4 и не чинились — это техдолг #2 Medium из S5
- Раздел «Счёт» T-Invest вернул балансы, но **позиции и операции остались заглушками** — техдолг #3 Medium из S5
- Новые процессные правила работы DEV-субагентов (`prompt_template.md`, Stack Gotchas, Integration Verification Checklist, Cross-DEV contracts) приняты сегодня, но **ни разу не применялись на практике** — надо обкатать на реальных задачах

Если начать Sprint 6 с этим багажом, он повторит паттерн Sprint 5: формальная реализация классов без runtime-интеграции + декларативные чеклисты без реальной проверки + красный CI, который никто не смотрит. Сначала надо стабилизироваться.

## Порядок работы

Рекомендуемая последовательность — по убыванию блокеров:

1. **CI cleanup (задача 1)** — **выполняется первым**. Без зелёного CI нельзя надёжно использовать Integration Verification Checklist из шаблона (grep-правила проверяют факт вызова, а не корректность, но если тесты не гоняются, мы просто не узнаем о регрессиях). Ориентировочный объём: 1-2 часа работы для `ruff per-file-ignores` + адаптация `react-hooks/*` правил + массовый фикс `no-unused-vars` через `eslint --fix`.

2. **E2E S4 fix (задача 4)** — параллельно или сразу после CI. Без работающих E2E нельзя валидировать ARCH-ревью ни S5R, ни S6. 30+ падающих тестов — это часть «тестирование работает или нет».

3. **Live Runtime Loop (задача 2)** — архитектурная задача, самая крупная. Требует нового модуля `app/trading/runtime.py`, поля `TradingSession.timeframe` + миграции Alembic, переделки `SignalProcessor._execute_strategy` (candles[] вместо одной свечи), UI-селектора таймфрейма, вывода TF в списке сессий, E2E теста «fake-стрим → сделка». Оценка: самая долгая из всех, минимум 2-3 рабочие сессии.

4. **Реальные позиции T-Invest (задача 3)** — независимая задача, можно делать параллельно Live Runtime Loop. Backend: реализовать `broker/router.py` через `OperationsService.GetPortfolio`/`GetOperations`, маппер. Frontend: подключение реальных данных к существующим таблицам.

5. **Актуализация ФТ/ТЗ (задача 6)** — финальная. После того как всё в коде сделано, обновляем `functional_requirements.md`, `technical_specification.md`, `development_plan.md`, `ui_checklist`. **Ретроспективно** за весь S5 + S5R — потому что на формальном закрытии S5 документация не была полностью синхронизирована.

6. **ARCH-ревью S5R** — финальная приёмка. Запускаем `arch_review_s5r.md`: сверка по всем 6 задачам, запуск полного тестового набора (backend pytest + frontend vitest + Playwright E2E), проверка Integration Verification для Live Runtime Loop, проверка, что CI зелёный на последнем коммите. Обязательная секция «Оценка эффективности нового процесса работы DEV-субагентов» — что сработало, что надо доработать в `prompt_template.md` и Stack Gotchas перед S6.
