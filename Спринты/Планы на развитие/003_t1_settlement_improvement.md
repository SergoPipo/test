# 003: Корректная реализация T+1 расчётов в Paper Trading

> **Приоритет:** Средний
> **Источник:** Обсуждение Sprint 5 — текущая реализация упрощена и не соответствует реальному поведению биржи
> **Дата:** 2026-04-08

---

## Проблема

Текущий механизм T+1 в Paper Trading имеет три существенных недостатка:

### 1. Ленивая разблокировка (нет фоновой задачи)

Средства разблокируются **не по истечении времени**, а только при следующем вызове `place_order()`:

```
place_order() → _auto_unblock_if_settled() → проверка: прошло 24 часа?
```

Если стратегия молчит 2 дня — средства формально заблокированы всё это время, хотя 24 часа давно прошли. Пользователь, глядя на баланс, видит некорректные цифры.

### 2. Единая дата для всех блокировок

Проверяется `portfolio.updated_at` — дата последнего обновления портфеля, а не дата конкретной сделки. Если было 3 покупки в разное время, **все** разблокируются одновременно по дате последней операции:

```
Покупка 1: понедельник 10:00 → должна разблокироваться во вторник
Покупка 2: понедельник 15:00 → должна разблокироваться во вторник
Покупка 3: вторник 12:00    → должна разблокироваться в среду

Сейчас: ВСЕ ТРИ разблокируются по дате покупки 3 (вторник 12:00 + 24ч)
```

### 3. Не учитывается торговый календарь

Текущая логика: `now - updated_at >= 86400` (ровно 24 часа).

На реальной бирже MOEX:
- Сделка в пятницу → расчёт в **понедельник** (не в субботу)
- Сделка перед праздниками → расчёт в **первый рабочий день** после
- Расчёт привязан к **торговым дням**, а не к астрономическим часам

---

## Предлагаемое решение

### 1. Отдельная таблица блокировок

Новая модель `FundBlock` вместо единого `blocked_amount`:

```
FundBlock:
  id
  portfolio_id       → FK на PaperPortfolio
  trade_id           → FK на LiveTrade (какая сделка вызвала блокировку)
  amount             → Decimal (заблокированная сумма)
  blocked_at         → DateTime (когда заблокировано)
  settlement_date    → Date (когда разблокировать — следующий торговый день)
  status             → 'blocked' | 'settled'
```

Каждая покупка создаёт свою запись блокировки с индивидуальной датой расчёта.

### 2. Фоновая задача разблокировки

APScheduler-задача, запускаемая по расписанию (каждые 5 минут или по торговому календарю):

```
Для каждого FundBlock где status = 'blocked' и settlement_date <= сегодня:
  → portfolio.balance += amount
  → portfolio.blocked_amount -= amount
  → status = 'settled'
  → Event Bus: "funds_settled" → WebSocket → фронтенд
```

### 3. Интеграция с торговым календарём MOEX

Используем уже существующий модуль `scheduler/` с календарём MOEX:

```
blocked_at = 2026-04-11 (пятница)
settlement_date = next_trading_day(2026-04-11) = 2026-04-14 (понедельник)
```

## Что затрагивает

### Backend (изменения)

| Файл | Изменения |
|------|-----------|
| `trading/paper_engine.py` | Убрать `_auto_unblock_if_settled()`, создавать `FundBlock` при покупке |
| `trading/models.py` | Новая модель `FundBlock` |
| `common/database.py` | Миграция: новая таблица `fund_blocks` |
| `scheduler/jobs.py` | Новая задача `settle_fund_blocks()` |
| `trading/service.py` | Учёт индивидуальных блокировок в отображении баланса |

### Backend (без изменений)

Существующие API-эндпоинты для баланса и позиций не меняются — меняется только логика расчёта `blocked_amount`.

### Frontend

Изменения минимальны — фронтенд уже показывает `blocked_amount`. Можно добавить:
- Tooltip на «Заблокировано» с расшифровкой: «Покупка SBER: 50 000 ₽, расчёт 14.04»

---

## Критерии готовности

- [ ] Каждая покупка создаёт отдельную запись `FundBlock`
- [ ] Фоновая задача разблокирует средства по `settlement_date`
- [ ] `settlement_date` вычисляется по торговому календарю MOEX (не +24ч)
- [ ] Баланс обновляется автоматически без ожидания следующего ордера
- [ ] WebSocket-уведомление при разблокировке
- [ ] Unit-тесты: блокировка в пятницу → разблокировка в понедельник
- [ ] Unit-тесты: 3 покупки в разное время → 3 разных даты расчёта

---

## Риски и ограничения

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Торговый календарь неточен | Праздничные дни могут отсутствовать | Парсить календарь с MOEX ISS или хранить вручную |
| Миграция данных | Существующие `blocked_amount` нужно конвертировать | Миграция: создать FundBlock из текущих LiveTrade |
| Производительность | Много записей FundBlock при активной торговле | Индекс по (portfolio_id, status), очистка settled записей |
