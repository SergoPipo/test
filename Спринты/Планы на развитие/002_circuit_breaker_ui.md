# 002: UI-панель настройки Circuit Breaker

> **Приоритет:** Средний
> **Источник:** Обсуждение Sprint 5 — CB работает, но настраивается только через API
> **Дата:** 2026-04-08

---

## Проблема

Circuit Breaker реализован на backend (9 проверок, JWT-авторизация, конфигурация на 4 уровнях), но **у пользователя нет визуального интерфейса** для:

- Просмотра текущих лимитов
- Изменения настроек (daily loss, max drawdown, торговые часы и т.д.)
- Просмотра истории срабатываний
- Понимания почему сессия ушла в паузу

Сейчас всё управление — через прямые API-запросы:
- `PUT /api/v1/circuit-breaker/config` — настройка
- `GET /api/v1/circuit-breaker/status` — текущее состояние
- `GET /api/v1/circuit-breaker/events` — история

Это неприемлемо для конечного пользователя.

---

## Предлагаемое решение

### Вариант: Страница «Управление рисками» (`/settings/risk`)

Доступна через sidebar (новый пункт меню) или через иконку ⚙ на странице `/trading`.

### Компоненты

| Компонент | Описание |
|-----------|----------|
| `RiskStatusPanel` | Карточки с текущими метриками (обновляются по WebSocket) |
| `RiskConfigForm` | Форма настройки глобальных лимитов |
| `RiskEventsTimeline` | Лента истории срабатываний с пагинацией |
| `SessionRiskOverride` | Модалка для переопределения лимитов конкретной сессии |

### Дополнительно на странице /trading

- На карточке сессии, которая ушла в паузу из-за CB: иконка ⚠ с tooltip «Приостановлена: Daily Loss Limit»
- Кнопка «Подробнее» → переход на `/settings/risk` с фильтром по этой сессии

### Frontend (изменения)

| Файл | Изменения |
|------|-----------|
| `components/layout/Sidebar.tsx` | Новый пункт меню «Риски» |
| `App.tsx` или `router.tsx` | Новый маршрут `/settings/risk` |
| `components/trading/SessionCard.tsx` | Иконка ⚠ при паузе из-за CB |

### Backend

Существующие API достаточны — новых эндпоинтов не требуется:
- `GET /api/v1/circuit-breaker/config` — загрузка настроек
- `PUT /api/v1/circuit-breaker/config` — сохранение
- `GET /api/v1/circuit-breaker/status` — статус
- `GET /api/v1/circuit-breaker/events` — история

Возможно потребуется:
- `GET /api/v1/circuit-breaker/events/session/{id}` — фильтр по сессии (уже есть)
- WebSocket канал для push-обновлений статуса (уже публикуется в Event Bus)

## Критерии готовности

- [ ] Страница `/settings/risk` доступна через sidebar
- [ ] Карточки статуса отображают актуальные данные
- [ ] Форма позволяет сохранять глобальные лимиты
- [ ] История срабатываний отображается с пагинацией
- [ ] Можно задать override для отдельной сессии
- [ ] На карточке сессии видно причину паузы CB
- [ ] Данные обновляются в реальном времени (WebSocket)
- [ ] E2E тест: изменить лимит → запустить сессию → проверить что лимит применяется

---

## Риски и ограничения

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Непонятные метрики | Пользователь не поймёт что такое «drawdown» | Tooltips с объяснениями + ссылка на FAQ |
| Опасные настройки | Слишком высокие лимиты = нет защиты | Предупреждения при значениях выше рекомендованных |
| Нагрузка на API | Частый polling статуса | Использовать WebSocket вместо polling |
