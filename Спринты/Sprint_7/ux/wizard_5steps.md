# UX-макет: First-run wizard 5 шагов (для задачи 7.8)

> **Статус:** заполнен UX-агентом на W0 (2026-04-25).
> **Источники:** `Sprint_7/sprint_design.md` Приложение D, `Sprint_7/prompt_UX.md` задача 2, контракт C6 в `Приложение A`.
> **Связь с задачей S7:** 7.8 «Onboarding wizard для новых пользователей».
> **Контракт:** C6 (BACK2 → FRONT2). Подтверждён в spec.

---

## 1. Описание сценария

При первом логине после регистрации (`wizard_completed_at IS NULL`) пользователь видит модальное окно `<FirstRunWizard>` поверх главной страницы. Окно — full-screen overlay (Mantine `<Modal fullScreen>` либо встроенная страница `/onboarding` с защитой роута). Закрыть нельзя кликом вне модалки или Esc — только пройти все 5 шагов. Шаг 2 (дисклеймер) — обязательный gate: без галки «Я прочитал» дальше нельзя.

После 5-го шага frontend вызывает `POST /api/v1/users/me/wizard/complete`, backend проставляет `wizard_completed_at = now()` и редиректит на `/dashboard` (виджет 7.7).

## 8. Поведение / навигация

| Событие | Реакция |
|---------|---------|
| Click `Далее` (валидно) | Step++ |
| Click `Назад` | Step-- (≥1) |
| Click `Завершить` (Step 5) | API call C6 → redirect `/dashboard` |
| Esc / Click outside | Игнор |
| Refresh страницы | Если `wizard_completed_at IS NULL` — wizard снова с шага 1 (или с последнего активного, если backend хранит progress; в S7 — с шага 1, не сохраняем in-progress). |
| Direct URL `/dashboard` без wizard | Редирект на wizard. |

## 9. Контракт с backend (C6)

**Поставщик:** BACK2. **Потребитель:** FRONT2.

- `GET /api/v1/users/me` → `{id, email, wizard_completed_at: datetime|null, ...}`. Frontend читает на mount → если null → показать wizard.
- `POST /api/v1/users/me/wizard/complete` body: `{broker: 'tinvest', mode: 'paper'|'real', notifications: {telegram?: {bot_token, chat_id}, email?: string, inapp: true}}` → 200, проставляет `wizard_completed_at`.
- `POST /api/v1/notifications/telegram/test` body: `{bot_token, chat_id}` → 200 ok / 4xx invalid (опционально, может быть в `prompt_DEV-1.md` BACK1 если уже есть).

`assume(ARCH)`: возможно бэкенд хочет сохранять каждое поле отдельно (`/users/me` PATCH с partial). UX готов к обоим вариантам — оборачиваем в один submit-action на frontend.

## 10. Доступность

- `<Modal>` с `aria-labelledby` и `aria-describedby`.
- Stepper — `aria-current="step"` для активного шага, `aria-label="Шаг N из 5"`.
- Чекбокс дисклеймера — `aria-required="true"`.
- Disabled `Далее` — `aria-disabled="true"` + `title` с подсказкой.
- Вся форма проходима только с клавиатуры (Tab, Shift+Tab, Enter).
- Высокий контраст текста на тёмной теме (≥ 4.5:1).
- Скролл внутри `<ScrollArea>` с клавиатуры (PgUp/PgDn).

## 11. Тестовые сценарии для QA

(в `e2e_test_plan_s7.md`, секция 7.8)

1. Новый пользователь после регистрации видит wizard на шаге 1.
2. Esc и клик вне — модалка не закрывается.
3. Шаг 2: `Далее` disabled до галки; после галки — enabled.
4. Шаг 3: переключение paper/real.
5. Шаг 4: добавление Telegram токена, тест-кнопка показывает success/error.
6. Шаг 5: `Завершить` → POST C6 → redirect `/dashboard`, повторный логин не показывает wizard.
7. Refresh страницы на любом шаге — снова шаг 1 (wizard ещё не complete).
8. После `wizard_completed_at` set — `/dashboard` доступен напрямую.

`data-testid`:
- `wizard-modal`, `wizard-stepper`.
- `wizard-step-1`...`wizard-step-5`.
- `wizard-disclaimer-checkbox`, `wizard-mode-paper`, `wizard-mode-real`.
- `wizard-telegram-token`, `wizard-telegram-chatid`, `wizard-telegram-test`.
- `wizard-button-next`, `wizard-button-back`, `wizard-button-finish`.
