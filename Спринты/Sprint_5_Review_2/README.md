# Sprint_5_Review_2 — Chart Hardening (патч-цикл после S5R wave 4)

> **Статус:** 🔄 в процессе — этап A (планирование, скелет)
> **Открыто:** 2026-04-16
> **Причина:** после закрытия S5R wave 4 (фазы 3.3–3.7) остались крупные треки, которые нельзя было уместить в рамки закрывающей волны — каждый требует отдельной Claude-сессии для экономии контекста.
> **Охватывает:** регрессии и ограничения модуля графиков (frontend `InstrumentChart`/`ChartPage` + backend `market_data`/`stream_manager`), выявленные при ручном тестировании 2026-04-15–16.

## Отличие от Sprint_5_Review

`Sprint_5_Review` (S5R) — внеплановое ревью по итогам Sprint 5, закрытое 2026-04-16 с вердиктом **PASS WITH NOTES**. В ходе wave 4 S5R были починены срочные регрессии графика (T-Invest naive→aware — `gotcha-15`, client-side `candlesCache` + persist, race-guard в `fetchCandles`, фикс D-мигания в `ChartPage`).

`Sprint_5_Review_2` (S5R-2) — **отдельный патч-цикл**, посвящённый оставшимся пяти крупным трекам chart-hardening. Это **не новый спринт** и **не расширение S5R** — это независимый мини-цикл, работающий параллельно с Sprint 6 (уведомления). Каждый трек — самостоятельный блок, который запускается в отдельной Claude-сессии.

## Почему созвано

После wave 4 S5R остались пять характерных классов проблем, выходящих за рамки «закрывающей волны»:

1. **Frontend prefetch через REST ограничен TF, рабочими часами и фрагментарностью** — при логине график пользователя медленно наполняется свечами, а для активных торговых сессий хотелось бы моментального старта.
2. **Daily/1h/4h свечи не обновляются в реальном времени** — T-Invest стрим шлёт 1-минутные тики, клиент агрегирует их только в `upsertLiveCandle`, бэкенд не создаёт производные таймфреймы поверх 1m.
3. **Intraday-график содержит визуальные «дыры»** в нерабочие часы MOEX — ось времени линейная, sequential-index mode не включён, при узком TF это выглядит как обрыв графика.
4. **401 Unauthorized после релогина** — сценарий воспроизводится неконсистентно, диагностика не завершена, подозрение на гонку `AuthContext` ↔ `useSWR` ↔ persist.
5. **Остаточное Daily-мигание после `3.7`** — race-guard в `fetchCandles` закрыл основную гонку, но singleton WebSocket может доставить тик в микро-интервал между store-update и unsubscribe. Требуется **TF-aware `upsertLiveCandle`**: фильтр тика по активному таймфрейму до того, как он дойдёт до store.

Все пять треков связаны модулем графика и разделяют общий тестовый контур (`ChartPage`, `InstrumentChart`, `market_data/router.py`, `stream_manager.py`). При этом они **независимы по файлам**: их можно выполнять параллельно в отдельных сессиях.

## Порядок работы

Треки **независимы по файлам** — допускается параллельное выполнение в отдельных Claude-сессиях (по одному треку на сессию). Рекомендуемая последовательность — по убыванию риска/блокирующего эффекта:

1. **Трек 4 (401 после релогина)** — первый, потому что UX-блокер: пользователь не может работать после перезахода без полного перезапуска.
2. **Трек 5 (TF-aware `upsertLiveCandle`)** — небольшой, закрывает остаточную гонку из wave 4, возвращает стабильность Daily-графика.
3. **Трек 1 (Backend prefetch)** — backend-feature, автономна.
4. **Трек 2 (Live-агрегация)** — крупнейшая, требует нового поддерева в `market_data/`, делается последней.
5. **Трек 3 (Sequential-index mode)** — чисто frontend, выполняется параллельно любому из backend-треков.

Финальное ARCH-ревью (`arch_review_s5r2.md`) — после закрытия всех 5 треков.

## Связь с Sprint 6

S5R-2 **не блокирует** Sprint 6. Sprint 6 (уведомления + recovery + graceful shutdown) стартует параллельно и может работать независимо. Пересечений по файлам в ожидаемых треках нет — S6 работает в `app/notifications/` и `app/trading/runtime.py`, S5R-2 — в `app/market_data/` и `frontend/src/features/chart/`.

Если при выполнении треков обнаружится регрессия в коде S6 — фиксируется в `Sprint_6/backlog.md` как отдельное замечание, не в S5R-2.

## Связь с правилами `prompt_template.md`

S5R-2 — **вторая итерация** практического применения `Спринты/prompt_template.md` после S5R. Если по итогам S5R-2 обнаружатся новые недостатки шаблона — фиксируются в `arch_review_s5r2.md` и вносятся в шаблон перед Sprint 6.
