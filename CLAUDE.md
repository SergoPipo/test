## Процесс разработки

### Спринтовая модель
- 8 спринтов, 4 майлстоуна (M1: Каркас → M2: Бэктест → M3: Торговля → M4: Production)
- Ревью после каждых 2 спринтов (Sprint_2_Review, Sprint_4_Review, ...)
- Текущий статус всегда в: Спринты/project_state.md

### Структура спринта
Каждый Sprint_N/ содержит:
- execution_order.md — порядок выполнения задач
- sprint_state.md — текущее состояние спринта
- prompt_ARCH_review.md — задание для ARCH-ревью
- prompt_DEV-1..N.md — задания для разработчиков
- prompt_UX.md — задание для UX
- changelog.md — лог изменений
- preflight_checklist.md — чеклист перед запуском

### Роли Claude-агентов
В prompt-файлах спринтов определены роли:
- **ARCH** — архитектурное ревью, проверка паттернов и решений
- **DEV-1..5** — реализация конкретных задач (backend/frontend)
- **UX** — дизайн интерфейса, спецификации UI

### Порядок работы со спринтом
1. Прочитать Спринты/project_state.md — понять где мы
2. Прочитать Sprint_N/execution_order.md — порядок задач
3. Прочитать Sprint_N/sprint_state.md — что уже сделано
4. Взять нужный prompt_*.md — выполнить задачу
5. Обновить changelog.md и sprint_state.md

### Работа с DEV-субагентами в спринтах

По итогам ретроспективы Sprint 5 (см. Sprint_5/changelog.md, запись от 2026-04-14) выявлены систематические слабости в формулировке промптов DEV-агентов, которые привели к предотвратимым багам и техдолгу (задачи 6.0 Live Runtime Loop и 6.9 Реальные позиции в S6). Правила ниже применяются **начиная с Sprint 6**:

- **Шаблон промпта** — при создании `Sprint_N/prompt_DEV-X.md` использовать `Спринты/prompt_template.md` как основу. Заполнять все 11 секций (роль, предпроверка, **обязательное чтение**, контекст, задачи, тесты, **integration checklist**, **формат отчёта**, alembic-миграции, чеклист).
- **Ловушки стека (Stack Gotchas)** — в секции «Обязательное чтение» промпта явно сослаться на `Develop/CLAUDE.md` раздел «⚠️ Stack Gotchas». Эта секция автоматически подгружается в системный контекст субагента при работе с `Develop/`. Каждое ARCH-ревью обязано дополнять её новыми ловушками, найденными в спринте.
- **Цитаты из ТЗ/ФТ** — не ограничиваться ссылками «ТЗ 5.4.2». Цитировать фрагменты дословно прямо в промпте, чтобы субагент видел архитектурное требование без дополнительных чтений.
- **Cross-DEV контракты** — в `Sprint_N/execution_order.md` обязательно заполнять раздел «Cross-DEV contracts» **до** запуска DEV-агентов. Таблица: поставщик | потребитель | контракт (endpoint/класс/метод) | формат/сигнатура. Каждый DEV в своём отчёте подтверждает контракты, в которых он — поставщик или потребитель. Несоответствие контракта — блокирующая ошибка для оркестратора.
- **Integration verification** — каждый DEV обязан в отчёте подтвердить, что новые классы/методы вызываются в production-коде (`grep -rn "<ClassName>(" app/`), не только в тестах. Если точка вызова не найдена — явно пометить `⚠️ NOT CONNECTED` в отчёте. Такая пометка блокирует формальную приёмку.
- **Формат отчёта субагента** — структурированная сводка до 400 слов по 8 секциям (реализовано / файлы / тесты / integration points / контракты / проблемы / применённые Stack Gotchas / новые Stack Gotchas). **Без** полного кода, **без** лога tool-вызовов.

## Правила
- Код, конвенции и архитектура описаны в Develop/CLAUDE.md — не дублировать здесь
- Перед началом работы — прочитать project_state.md и changelog текущего спринта
- Настроечные файлы Claude Code (.claude/) — не коммитить в git
- .env, ключи API, токены — никогда не коммитить

## E2E тесты — обязательный процесс

«Создай E2E тесты» = полный цикл, а не просто файлы:

1. **Описание** — ПЕРЕД написанием тестов создать `e2e_test_plan_sN.md`: сценарии, предусловия, шаги, ожидаемый результат. Основа — `ui_checklist` + новый функционал спринта.
2. **Инфраструктура** — в `preflight_checklist.md`: проверить Playwright (`npx playwright --version`, `playwright.config.ts`). В `execution_order.md`: шаг «поднять backend + frontend».
3. **Написание** — тесты по спецификации из п.1.
4. **Запуск** — `npx playwright test`. Тесты ОБЯЗАНЫ быть реально выполнены, не просто написаны.
5. **Верификация** — результат прикладывается в `arch_review_sN.md`. Критерий: passed/failed, а не «файлы созданы».
